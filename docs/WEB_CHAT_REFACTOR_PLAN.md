# WEB CHAT REFACTOR PLAN

## 1. 目标

把当前 `ngent` 从“多 Agent 管理面板”改造成“网页端对话产品”。

新的产品形态应当更接近常见的 Chat UI，而不是运维/调试工具：

- 用户打开网页即可直接开始对话
- 主要心智模型从“线程 + Agent 管理”变成“会话式聊天”
- 普通用户优先，不要求先理解 ACP、CWD、权限卡片等内部概念
- 保留现有后端流式、持久化、权限审批能力，但在 UI 上做产品化包装

这次改造的首要目标不是重写后端，而是：

1. 先重构前端信息架构和交互模型
2. 在尽量复用现有 API 的前提下完成网页端聊天体验
3. 再逐步补齐为聊天产品服务的后端接口

## 2. 当前问题

当前项目已经有可用的 Web UI，但它更像“Agent 控制台”，存在这些问题：

- 首页信息密度偏高，普通用户上手成本高
- “新建 Agent / 工作目录 / 高级 JSON”暴露了太多工程内部概念
- 会话列表、线程设置、权限、Slash Commands 等能力是平铺的，缺少主次
- `GET /v1/agents/{agentId}/models` 与线程级 `config-options` 存在体验割裂
- UI 的目标是“能操作系统能力”，不是“自然连续地聊天”

结论：后端能力已经足够支撑聊天产品雏形，但前端产品定义还没有收敛。

## 3. 改造后的产品定义

### 3.1 产品定位

一个部署在浏览器中的 AI 对话工作台，支持：

- 连续多轮对话
- 基于会话的上下文记忆
- 可选模型切换
- 文件上传与图片粘贴
- 流式回复
- 权限审批
- 面向项目目录的代码/任务协作

### 3.2 核心用户路径

1. 用户打开网页
2. 看到聊天首页和历史会话列表
3. 点击“新对话”
4. 输入问题并开始对话
5. 在必要时切换模型、审批权限、查看会话历史
6. 上传文件或直接粘贴截图参与对话

### 3.3 设计原则

- 聊天优先：输入框、消息流、历史列表是一级界面
- 配置后置：CWD、Agent、模型、权限策略收进次级入口
- 渐进暴露：默认只展示用户当下必须理解的内容
- 移动兼容：优先支持桌面，其次保证手机可读可操作
- 可运维：不牺牲现有日志、线程、权限、SSE 能力
- 容量有边界：聊天附件和衍生文件总空间必须可控

## 4. 目标界面结构

### 4.1 主界面

采用标准 Chat App 布局：

```text
┌──────────────────────────────────────────────────────────────┐
│ Sidebar                       │ Chat                         │
├───────────────────────────────┼──────────────────────────────┤
│ Logo / App Name               │ Chat Header                  │
│ [ + 新对话 ]                  │ 标题                         │
│ 搜索会话                      │ 模型切换                     │
│ 历史会话列表                  │ 会话菜单                     │
│                               │------------------------------│
│                               │ 消息流                       │
│                               │ 用户消息                     │
│                               │ 助手消息                     │
│                               │ 权限审批卡片                 │
│                               │ 流式生成状态                 │
│                               │------------------------------│
│                               │ 输入框 + 发送按钮            │
└───────────────────────────────┴──────────────────────────────┘
```

### 4.2 主要区域

- 左侧边栏
  - 新对话按钮
  - 历史会话列表
  - 搜索会话
  - 设置入口
- 右侧聊天区
  - 顶部显示当前会话标题、模型、更多操作
  - 中部为消息流
  - 底部为输入框
  - 输入框上方显示待发送附件区

### 4.3 弱化或后移的能力

这些能力保留，但不再放在默认主路径：

- CWD 输入
- Agent 切换
- 原始 JSON 配置
- Slash Commands 浏览
- Session transcript 调试视图

默认主路径应新增两项能力：

- 上传文件
- 粘贴图片

建议放到：

- “新对话”弹窗的高级设置
- 当前会话右上角“设置”抽屉
- 开发者模式下的高级面板

## 5. 信息架构调整

### 5.1 顶层对象映射

现有后端对象与新产品术语建议映射如下：

| 当前概念 | 新 UI 命名 | 说明 |
|---|---|---|
| thread | 对话 / 会话 | 主用户心智对象 |
| turn | 一轮消息 | 一次用户输入及对应回复 |
| agent | 提供者 | 默认隐藏，仅在高级设置可见 |
| cwd | 工作区 | 默认有值，非必填暴露 |
| agentOptions.modelId | 模型 | 暴露给用户 |
| permission_required | 操作确认 | UI 用自然语言呈现 |
| uploaded file / pasted image | 附件 | 作为消息的一部分展示 |

### 5.2 首页默认行为

建议首页不再要求用户先创建带 CWD 的线程，而是：

- 点击“新对话”即可创建会话
- 默认工作区使用服务端预设目录
- 默认 Agent 使用 `codex`
- 默认模型来自服务端或用户最近一次选择

### 5.3 新对话弹窗

默认只显示：

- 对话标题，可选
- 模型，可选

折叠“高级设置”后才显示：

- 工作区
- Agent
- 原始参数

聊天输入区默认直接支持：

- 拖拽文件
- 点击上传
- `Ctrl+V` / `Cmd+V` 粘贴图片

## 6. 附件与存储配额

### 6.1 用户需求

聊天产品需要支持：

- 上传文件
- 直接粘贴图片

同时需要控制磁盘占用：

- 聊天产生的总空间上限为 `5GB`
- 超过上限后，按时间顺序销毁更早的数据

### 6.2 建议的产品行为

#### 文件上传

支持用户上传这些类型：

- 文本文件
- 代码文件
- PDF
- Office 文档
- 压缩包
- 常见图片格式

单条消息可附带多个文件，但应限制：

- 单文件大小上限
- 单次消息总附件大小上限
- MIME 类型白名单

#### 粘贴图片

支持从系统剪贴板直接粘贴：

- 截图
- 剪贴板中的 PNG / JPEG / WebP 图片

前端行为建议：

- 粘贴后先显示缩略图预览
- 用户确认发送后才真正创建消息
- 可在发送前删除某个附件

#### 消息展示

消息流中的附件应以卡片形式展示：

- 图片显示缩略图，可点击放大
- 非图片显示文件名、大小、类型、下载按钮
- 每条消息清楚标记哪些附件是用户上传的

### 6.3 存储口径

“聊天产生的空间”建议包含：

- 用户上传的原始文件
- 粘贴图片落盘后的二进制文件
- 对附件生成的缩略图
- 模型生成并持久化的导出文件
- 与消息关联的中间产物

建议默认不把这些内容计入：

- SQLite 主数据库本身
- 可重建的缓存
- 临时流式内存状态
- 代码仓库本体文件

原因：

- 否则会把系统运行基础数据和用户产物混在一起
- 5GB 配额应主要约束“聊天资产”，不是整个服务目录

### 6.4 配额策略

建议按“聊天资产总量”执行全局配额控制：

- 总上限：`5GB`
- 水位触发：写入前检查，写入后再次校验
- 清理顺序：按最早未访问或最早创建的数据优先淘汰

建议默认采用：

- 主规则：按消息/附件 `created_at` 升序淘汰
- 辅规则：同一消息下的关联文件一起删除，避免孤儿记录

淘汰原则：

1. 先删除最旧会话中的附件文件
2. 若某会话附件全部清空，可继续删除其历史消息与 turn 数据
3. 若仍超限，再删除整个最旧会话

这样更符合你的要求：

- “超出就销毁之前的”

### 6.5 建议的删除粒度

为了避免数据关系混乱，建议按下面粒度删除：

- 第一优先级：附件文件
- 第二优先级：空附件消息的附件记录
- 第三优先级：整条消息 / turn
- 第四优先级：整段会话 / thread

不建议只删文件、不删元数据，也不建议只删单个 turn 的一部分附件而不更新消息内容。

### 6.6 服务端落盘目录

建议新增统一资产目录，例如：

- `$HOME/.ngent/assets/`

目录结构建议：

```text
$HOME/.ngent/assets/
  threads/
    th_xxx/
      2026/03/17/
        file_xxx.pdf
        image_xxx.png
        thumb_xxx.webp
```

要求：

- 文件名去随机化冲突
- 不信任用户原始文件名作为真实落盘名
- 元数据里保留原始文件名用于展示

### 6.7 数据库设计建议

建议新增两张表。

#### `message_attachments`

- `attachment_id TEXT PRIMARY KEY`
- `thread_id TEXT NOT NULL`
- `turn_id TEXT`
- `role TEXT NOT NULL`
- `kind TEXT NOT NULL`
  - `file|image`
- `origin_name TEXT NOT NULL`
- `stored_name TEXT NOT NULL`
- `mime_type TEXT NOT NULL`
- `size_bytes INTEGER NOT NULL`
- `storage_path TEXT NOT NULL`
- `thumbnail_path TEXT NOT NULL DEFAULT ''`
- `sha256 TEXT NOT NULL DEFAULT ''`
- `created_at TEXT NOT NULL`
- `last_accessed_at TEXT NOT NULL`
- `deleted_at TEXT NOT NULL DEFAULT ''`

#### `storage_quota_state`

- `scope TEXT PRIMARY KEY`
  - 先用固定值 `global`
- `used_bytes INTEGER NOT NULL`
- `max_bytes INTEGER NOT NULL`
- `updated_at TEXT NOT NULL`

### 6.8 API 建议

建议新增这些接口：

#### A. 上传附件

`POST /v1/uploads`

请求：

- `multipart/form-data`

返回：

```json
{
  "uploads": [
    {
      "uploadId": "up_...",
      "name": "design.png",
      "mimeType": "image/png",
      "sizeBytes": 123456,
      "kind": "image"
    }
  ]
}
```

#### B. 发消息时关联附件

`POST /v1/threads/{threadId}/turns`

请求体新增：

```json
{
  "input": "请分析这张图",
  "stream": true,
  "attachments": ["up_123", "up_456"]
}
```

#### C. 查询存储状态

`GET /v1/storage`

返回：

```json
{
  "maxBytes": 5368709120,
  "usedBytes": 2147483648,
  "usagePercent": 40,
  "policy": "delete_oldest_first"
}
```

#### D. 下载附件

`GET /v1/attachments/{attachmentId}`

#### E. 删除会话资产

`DELETE /v1/threads/{threadId}/assets`

用于手动释放空间。

### 6.9 清理任务

建议增加后台清理器：

- 每次成功上传后触发一次快速检查
- 服务启动时做一次配额扫描
- 周期性巡检，例如每 5 分钟一次

清理流程建议：

1. 读取当前资产总大小
2. 若未超过 5GB，直接返回
3. 按淘汰顺序列出最旧资产
4. 分批删除，直到低于阈值
5. 更新配额状态和清理日志

### 6.10 用户体验建议

当接近上限时，前端应有提示：

- 超过 80% 显示“空间即将用满”
- 超过 95% 显示“旧附件可能被自动清理”
- 发生实际清理后，在系统消息中提示：
  - “为释放空间，系统已清理最早的历史附件/会话”

### 6.11 安全与风控

上传与图片粘贴需要补这些约束：

- MIME 白名单校验
- 服务端二次校验真实文件类型
- 文件大小限制
- 病毒扫描预留扩展点
- 图片 EXIF 脱敏或可选清洗
- 禁止可执行文件直接作为可访问静态资源运行

## 7. 后端复用与接口演进

### 6.1 当前可直接复用的接口

- `GET /v1/agents`
- `GET /v1/agents/{agentId}/models`
- `POST /v1/threads`
- `GET /v1/threads`
- `GET /v1/threads/{threadId}`
- `GET /v1/threads/{threadId}/history`
- `POST /v1/threads/{threadId}/turns`
- `POST /v1/turns/{turnId}/cancel`
- `POST /v1/permissions/{permissionId}`
- `GET /v1/threads/{threadId}/config-options`
- `POST /v1/threads/{threadId}/config-options`

### 6.2 建议新增或调整的接口

为让前端真正像聊天产品，而不是管理台，建议补这些能力：

#### A. 默认会话创建配置

新增一个服务端默认配置接口，例如：

`GET /v1/app/bootstrap`

返回：

```json
{
  "defaultAgent": "codex",
  "defaultModelId": "gpt-5.4",
  "defaultCwd": "/home/ubuntu/project/ngent",
  "features": {
    "multiAgent": false,
    "developerMode": true
  }
}
```

作用：

- 减少前端硬编码
- 让首页能直接“开始聊天”
- 支持不同部署环境使用不同默认值

#### B. 会话摘要字段增强

当前会话列表主要依赖标题和更新时间，建议后续加入：

- 最后一条消息摘要
- 最后发送者
- 未完成状态

这样侧边栏更像真正聊天产品。

#### C. 模型列表来源统一

当前存在两个来源：

- agent 级模型列表
- thread 级 config-options

建议统一规则：

- 新建会话时优先展示默认模型列表
- 已有会话一律以 `config-options` 为准

## 8. 前端实施方案

### Phase 1：先做“能聊”的聊天界面

目标：

- 页面一打开就是聊天产品形态
- 可新建会话、发消息、流式显示、看历史

任务：

- 重构首页布局，去掉明显的运维感
- 收敛新对话入口
- 聊天区强调消息流和输入框
- 侧边栏改成“历史会话”
- 模型切换收进顶部栏或弹窗
- 文件上传和图片粘贴先打通基础链路

### Phase 2：补产品化体验

目标：

- 让 UI 更像成熟 Chat 产品

任务：

- 会话搜索
- 最近会话分组
- 空状态引导
- 首屏欢迎页
- 更好的流式动效
- 权限卡片文案重写
- 附件预览、上传进度、图片灯箱
- 存储使用量提示

### Phase 3：补开发者/高级模式

目标：

- 不丢失 `ngent` 原本的工程能力

任务：

- 高级设置抽屉
- 工作区切换
- Agent 切换
- Slash Commands 面板
- Session / transcript 调试视图
- 存储清理策略与配额配置面板

## 10. 风险与注意事项

### 9.1 不要一次性推翻现有后端模型

当前线程/turn/SSE/权限链路已经稳定，前端重构应优先复用这些基础设施。

### 9.2 避免把“配置能力”继续暴露成首页主路径

如果仍把 CWD、Agent、JSON 配置放在用户第一屏，这次改造会停留在“换皮”。

### 9.3 模型展示要允许手动输入

鉴于当前 `codex` 模型列表可能和实际可用模型存在时差，前端仍应保留手动指定模型 ID 的能力。

### 10.4 权限审批需要更产品化文案

不要直接展示协议字段，应该转换成用户能理解的说明：

- 将执行命令
- 将读取文件
- 将访问网络

### 10.5 5GB 配额必须定义“删什么”

如果不先定义清楚“聊天产生的空间”口径，后续实现很容易出现：

- 删掉附件但数据库还在
- 删掉消息但文件没删
- 把系统关键数据误算进 5GB

所以配额实现必须先定：

- 统计范围
- 删除顺序
- 删除粒度
- 用户提示

### 10.6 图片粘贴和上传不是同一条链路

前端入口可以统一，但服务端最好统一成“上传资产”模型，不要分别造两套存储逻辑。

## 11. 验收标准

完成这轮改造后，至少应满足：

- 用户打开网页后，无需理解 Agent/CWD，即可开始新对话
- 新对话路径不超过 2 次点击
- 主界面核心区域是“消息流 + 输入框”
- 会话列表体现聊天产品形态，而不是线程调试列表
- 模型切换可用，且支持手动指定模型 ID
- 支持上传文件和粘贴图片参与对话
- 聊天资产总空间超过 `5GB` 后，系统会自动清理最旧历史
- 现有流式输出、权限审批、历史持久化能力保持可用

## 12. 下一步建议

建议按下面顺序推进：

1. 先确定聊天产品的页面结构和交互稿
2. 再改现有前端组件树
3. 最后补 `bootstrap` / 默认配置类接口

如果进入实现阶段，建议优先改这些文件：

- `internal/webui/web/src/main.ts`
- `internal/webui/web/src/style.css`
